      • allowCredentials

        public CorsConfigBuilder allowCredentials()
        默认情况下，Cookie不包含在CORS请求中，但是这种方法将启用cookies以添加到CORS请求中。 调用这个方法会将CORS的Access-Control-Allow-Credentials响应头设置为true。 请注意，该cookie支持也需要在客户端启用。 客户需要选择通过以下方式发送cookie： 
          xhr.withCredentials = true;
        'withCredentials'的默认值为false，在这种情况下，不发送cookie。 如果将此设置为true，则会在交叉源请求中包含Cookie。
        结果
        CorsConfigBuilder支持方法链接。

      • maxAge

        public CorsConfigBuilder maxAge(long max)
        当进行预检请求时，客户端必须执行两个请求，效率可能很低。 此设置将设置CORS的“Access-Control-Max-Age”响应标题，并启用指定时间的预检响应缓存。 在此期间，不会进行预检请求。
        参数
        max - 可以高速缓存预检响应的最长时间（以秒为单位）。
        结果
        CorsConfigBuilder支持方法链接。

      • allowedRequestHeaders

        public CorsConfigBuilder allowedRequestHeaders(java.lang.String... headers)
        指定CORS'Access-Control-Allow-Headers'响应头中应该返回的if头。 如果客户端在请求中指定了标头，例如通过调用： 
          xhr.setRequestHeader('My-Custom-Header', "SomeValue");
        服务器将在预检请求的'Access-Control-Request-Headers'中接收上述标题名称。 然后，服务器将决定是否允许为真正的请求发送此头（请记住，预检不是真正的请求，而是请求服务器是否允许请求）。
        参数
        headers - 要添加到预检'Access-Control-Allow-Headers'响应标题中的标题。
        结果
        CorsConfigBuilder支持方法链接。

      • preflightResponseHeader

        public <T> CorsConfigBuilder preflightResponseHeader(java.lang.CharSequence name,
                                                     java.util.concurrent.Callable<T> valueGenerator)
        返回应该添加到CORS预检响应中的HTTP响应标头。 像负载均衡器这样的中介可能会要求CORS预检请求设置了某些标题。 这使得可以添加这样的头部。 一些值必须在创建HTTP响应时动态创建，例如'Date'响应头。 这可以通过使用Callable来完成，当创建HTTP响应时，Callable将调用其“调用”方法。

      • shortCircuit

        public CorsConfigBuilder shortCircuit()
        指定CORS请求在进一步处理之前应该被拒绝。 在处理请求后设置CORS头。 这可能并不总是需要的，这个设置将检查Origin是否有效，如果它不是有效的，则不会进行进一步的处理，并且错误将被返回给呼叫客户端。
        结果
        CorsConfigBuilder支持方法链接。
